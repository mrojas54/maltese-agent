# Tech Debt Audit — maltese-agent

**Date:** 2026-07-10
**Scope:** `falcon-mcp` (Rust MCP server), `falcon-detective` (TypeScript Barnum workflow), `falcon-agent` (deliberately-broken Rust service), CI, docs. ~6,300 LOC, dormant since 2026-05-04, with the Rust NYC workshop as the next consumer.
**Method:** structural survey + three parallel component audits, findings spot-verified. Scored on Impact (1–5), Risk (1–5), Effort (1–5); Priority = (Impact + Risk) × (6 − Effort).

## Framing

`falcon-agent`'s poisoned prompt, missing guards, planted lints, and `#[ignore]`'d test are **intentional workshop material** — none of that is counted as debt. What *is* debt is anything that threatens the workshop's reliability on stage or misleads a maintainer about which defects are deliberate.

A pointed irony shapes this list: the project teaches schema validation, retry discipline, and sandbox guardrails, yet its own tooling has `z.any()` escape hatches, unchecked `as T` casts on tool results, and a sandbox whose advertised per-tool timeouts only exist for one of 19 tools.

## Business justification, condensed

- **TD-1/TD-2** are trust-destroyers: a test suite that can delete uncommitted work or report green without running is worse than no suite — it will be relied on during last-minute pre-workshop edits, which is exactly when it bites.
- **TD-3** threatens the product itself. This repo's product is *pedagogy*: if the marker points at the wrong import, the detective agent (or a helpful human) "fixes" the wrong thing and the workshop narrative breaks on stage.
- **TD-4** is the live-demo killer: the investigation "runs in 6–8 minutes on stage" only if no `cargo test` or git call ever hangs. One stall = 15 minutes of dead air before the client timeout fires.
- **TD-6/TD-7/TD-5** are the reliability debts — the exact class of failure the workshop teaches people to prevent.
- **TD-8/TD-9/TD-13** are cheap one-time infra installs that stop future debt accruing silently.

## Phased remediation plan

### Phase 0 — before anyone runs `npm test` again (~half day)
- TD-1: guard the checkout — refuse or stash if `falcon-agent/` is dirty
- TD-2: fail or explicitly `skip` with reason instead of silent pass
- TD-3: move the marker to the real dead import; rewrite the `#[ignore]` reason to say "intentional workshop smoking-gun — see README"; flag the poisoned example in `prompt.rs`; document lint #4 or remove it
- TD-13: add `rust-toolchain.toml`

### Phase 1 — guardrail parity (~1–2 days)
- TD-4: one `tokio::time::timeout` wrapper around all child-process tools (pairs with the TD-14 shared-subprocess helper)
- TD-5: real zod schemas for `proposePoisonFix`
- TD-7: zod-validate MCP results at the client boundary
- TD-8 + TD-9: Biome or ESLint+Prettier; `cargo audit`/`npm audit` CI steps; dependabot config
- TD-12: one shared model-constants module

### Phase 2 — resilience (~2–4 days)
- TD-6: pin `@barnum/barnum` exactly and/or get off `__definition.handle` — highest silent-breakage risk in the repo
- TD-10: unit tests for `verify`, `commit`, `finalSweep`, then `processIssues` with a mocked MCP client
- TD-11: resolve allowlisted binaries to absolute paths at startup
- TD-14: dedupe the propose* handlers and the search pipeline

### Phase 3 — post-workshop only
- TD-15, TD-16. **Deliberately deferred:** upgrading `@google/genai` (and possibly vitest) risks changing request shapes, which invalidates the recorded cassettes the e2e strategy depends on — do it after the workshop with a cassette re-record, not before.
