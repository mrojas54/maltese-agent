# MA-37: Share CARGO_TARGET_DIR in e2e to avoid cold run-worktree compile

OUT OF CONTRACT (MA-29 lineage, no AC IDs). Root-caused 2026-07-11 during MA-31/MA-32. The full e2e replay's wall-clock is dominated by a COLD cargo compile of falcon-agent inside the isolated .runs/e2e-test worktree. Two cargo surfaces, both cold when CARGO_TARGET_DIR is unset: (1) pipeline cargo via falcon-mcp — src/lib/mcp.ts:56 forwards CARGO_TARGET_DIR only if already set in the detective env; (2) post-pipeline smoke 'cargo test' at tests/e2e.test.ts:168 passes no env. Evidence (this machine, one variable): cold target ~530s (exceeds the 360s testTimeout MA-29 set) vs warm shared target =<repo>/target ~318s PASS with zero cassette misses; the instrumented warm run created no .runs/e2e-test/target, proving the forward path works and is simply unwired. Not a regression from the migrations (cargo path untouched, behavior-neutral); CI stays green because Swatinem/rust-cache + the cargo registry cache make the cold compile fit the budget (ci.yml:149-155 documents the cold-target cost knowingly). Impact: local 'npm run test:full' races the budget unless the dev exports CARGO_TARGET_DIR. Fix: default CARGO_TARGET_DIR to <REPO_ROOT>/target (already warm from the Build falcon-mcp step) for BOTH the pipeline CLI env and the smoke cargo test in tests/e2e.test.ts; the run worktree then reuses artifacts (~530s->~230s locally, CI headroom).
